## Problem Statement

PBV form documents are generated by stamping intake data onto blank source PDFs using a per-form **field map** (coordinates + font sizes). The field maps were authored by hand against PDF coordinate space, and several produce **spacing/placement defects** â€” overlapping fields, text that runs past its box, values landing in the wrong region. The only way to catch these today is to generate a document and eyeball all 11 forms, which does not scale and which the operator does not have the patience to do per applicant.

The defect lives in the **field map**, not in any individual generated PDF, so editing signed PDFs is the wrong fix (and, for HUD/HACH legal forms, a compliance problem). The right fix is to correct the field maps once and regenerate â€” and, going forward, to author field maps for **new documents** through a tool that proves placement visually before it is committed.

What the operator wants: **upload a document â†’ the tool does a few passes to propose where each field goes â†’ see a grayed-out preview (field boxes, no text) â†’ see a text-filled preview â†’ approve.** The approved output is the field map the existing generator consumes.

## Core Features

### 1. Upload + ingest
Accept a source PDF. Record page count and each page's media box (the bounds all placement is validated against).

### 2. "A few passes" â€” proposed field map
- **Pass 1 â€” anchor detection.** Extract the text layer and detect fill targets: underscore runs, ruled lines, empty boxes, and label tokens (e.g. "Name", "Date", "SSN"). Produce candidate field regions with (page, x, y, width, height).
- **Pass 2 â€” labelâ†’data-key mapping.** Match detected labels against the `pbv-field-inventory.md` conventions and `field-mapping.ts` resolvers to assign each candidate a `name`/data key and a sensible `font_size`. Unmapped candidates are flagged `fresh_input` (left blank) rather than guessed.
- **Pass 3 â€” geometry fit.** Snap font size and box width so representative values fit; detect repeating tables and emit `row_patterns` (`row_start_y`, `row_pitch`, `max_rows`, `columns`) rather than one field per cell.
- Output is a draft `FieldMap` (the exact `stamper.ts` shape).

### 3. Preview A â€” grayed-out write spacing
Render the source PDF with a translucent rectangle at every proposed field/row-cell box (no text). This is the spacing proof: overlaps, out-of-bounds boxes, and cramped rows are visible at a glance. Overlaps and out-of-bounds boxes are tinted a distinct warning color and listed.

### 4. Preview B â€” text-filled
Run `stampForm()` with a representative sample dataset (a configurable fixture; for PBV, derive from an existing intake snapshot). This is the exact production render. The operator sees real text in real positions.

### 5. Automated validation (the "OCR finalize" the operator asked for)
- **Geometric:** field rect within page media box; no overlap between field rects (configurable tolerance); `row_patterns` do not overflow available vertical space (`row_start_y âˆ’ row_pitch Ă— max_rows â‰¥ page bottom`).
- **OCR round-trip:** OCR Preview B and, per field region, compare extracted text against the sample value that was stamped there. Mismatch / missing / truncated text is flagged with the form_id, field name, page, and box. This closes the loop on "is each form actually correct" without a human reading every page.
- Validation output is a per-document report: pass, or a list of flagged fields with reasons.

### 6. Approve â†’ emit artifact
On approval, persist the approved `FieldMap`. **Phase A:** write/update the repo field-map config consumed by the generator. **Phase B:** store as a reusable, versioned template record (see Data Model) usable for any future document.

## Acceptance / verification

- For each of the 11 PBV forms: Preview A shows no overlap/out-of-bounds boxes; Preview B OCR round-trip matches the sample dataset per field (no truncation/misplacement flags).
- A deliberately broken field map (overlapping boxes, a box past the page edge, a row pattern that overflows) is flagged by the validators, not silently passed.
- Preview B is byte-for-byte the same render path as production generation (both call `stampForm`) â€” confirmed by generating one document through the live pipeline and through the tool with the same data and diffing.
- Phase B: an approved map in `document_field_maps` is the one the generator uses; a new upload can be taken from blank PDF to approved map entirely in the admin UI.
- [Inference] The OCR round-trip will catch value-placement defects; it is a heuristic over rendered text and is not asserted to catch every visual nuance (e.g. subtle baseline shifts within an otherwise-correct box) â€” those remain caught by the human approve step. OCR behavior is not guaranteed.

## Non-goals

- No editing of already-signed PDFs (compliance â€” fix the map and regenerate instead).
- No change to the signing ceremony, intake, or notification flow (PRD-85 owns notifications).
- No auto-approval â€” a human always approves before a map becomes `approved`.
- No external/3rd-party document service; OCR and rendering stay server-side over local bytes.
